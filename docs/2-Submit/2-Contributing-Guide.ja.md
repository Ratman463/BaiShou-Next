# コントリビューション方針

[简体中文](./2-Contributing-Guide.md) | [繁體中文](./2-Contributing-Guide.tw.md) | [English](./2-Contributing-Guide.en.md) | [日本語](./2-Contributing-Guide.ja.md)

**対象読者**：BaiShou-Next に参加したいすべての人間のコントリビューターと AI アシスタント。

本文では **歓迎する貢献の種類**、**事前に議論が必要なこと**、機能提案 Issue に含めるべき内容を説明します。  
提出・CI は [1-Submit-Rule.md](./1-Submit-Rule.md)（中国語）、コーディング規約は [1-AI-Code-Rule.md](../1-AI-Code/1-AI-Code-Rule.md)（中国語）を参照してください。

---

## 1. 私たちの立場（まずお読みください）

白守はプライバシーを重視した AI 記憶コンパニオンです。**製品の方向性・UX・データモデル**は長期的かつ慎重な計画が必要であり、外部 PR で機能を積み上げるものではありません。

| 種類 | 私たちの姿勢 |
| ---- | ------------ |
| **バグ修正**（再現可能・テストあり・CI 通過） | **歓迎** |
| **ドキュメント修正**（誤字・不明瞭な表現） | **歓迎** |
| **再現可能な Bug Issue**（環境・手順・期待/実際） | **歓迎** |
| **事前議論のない新機能 PR** | **先に Issue で議論；事前合意がない場合はマージされにくい** |
| **新 UI ライブラリ / 依存追加 / Schema 変更** | **先に Issue・メンテナ承認必須** |

> **要約**：「どう実装するか」の前に「やるべきか」を確認してください。  
> 大きな PR より、動機と方針が明確な Issue の方が受け入れられやすいです。

## 2. 最も歓迎する貢献：バグ修正

### 2.1 Bug Issue の作成

[バグ報告](https://github.com/Anson-Trio/BaiShou-Next/issues/new?template=01-bug.yml) テンプレートを使用し、可能な限り以下を含めてください。

- 白守のバージョン / ブランチ、デスクトップまたはモバイル、OS
- **再現手順**（クリーンな状態から）
- **期待される動作** vs **実際の動作**
- ログ・スクリーンショット（秘密情報・API Key はマスク）

### 2.2 バグ修正 PR

1. 対応 Issue で議論するか、PR 説明に Issue をリンク。
2. [1-AI-Code-Rule.md](../1-AI-Code/1-AI-Code-Rule.md) に従う：**一行の hotfix を除きテスト必須**。
3. PR 前に `pnpm ci:check` を実行（[1-Submit-Rule.md](./1-Submit-Rule.md)）。
4. 根本原因・修正方針・検証方法を記載。

範囲が小さく、動機が明確で、テストが十分なバグ修正 PR を最優先でレビュー・マージします。

## 3. 新機能：先に Issue、コードはその後（よく考えて）

### 3.1 新機能を先に議論する理由

- 白守の核は「記憶の伴侶」とローカルプライバシーです。機能は **データモデル・AI ツールチェーン・マルチプラットフォーム整合・バックアップ互換** などに波及し、ボタンを一つ足すだけでは済みません。
- 本 monorepo は **Electron デスクトップ** と **Expo モバイル** を同時に保守しています。多くの変更は両端の整合、または単一端のみの明確な理由が必要です。

**メンテナと合意する前に送られた Feature PR は、レビューなしでクローズされる場合があります。**

### 3.2 それでも新機能を提案したい場合

**Issue のみ**を開いてください（[機能提案](https://github.com/Anson-Trio/BaiShou-Next/issues/new?template=02-enhancement.yml) テンプレート）。**大きな PR を先に出さないでください。**

AI で Issue の下書きは構いませんが、**投稿前に必ず自分で読み、内容の正確さと範囲を確認**してください。未検証の AI 出力ではなく、あなたの判断が必要です。

#### Issue に含める項目

```markdown
## 機能提案

### 1. 何を解決するか？
（ユーザーシナリオ、課題；白守の「記憶コンパニオン / ローカルプライバシー」との関係）

### 2. 提案する解決策（ユーザーから見える動作）
（操作イメージ、入口、既存機能との重複）

### 3. 技術概要
- **変更予定のパッケージ/パス**（下記 monorepo マップ参照）
- **DB Schema への影響**（あり/なし；ありの場合は移行計画）
- **デスクトップ / モバイル / 両方**
- **新規依存**（名前、サイズ、既存スタックで足りない理由）

### 4. UI とコンポーネント
- 新 UI？`packages/ui` を再利用か新規作成か？
- **サードパーティ UI ライブラリを入れる場合**：名前、バージョン、ライセンス、[UI テーマ規約](../1-AI-Code/2-UI-Theme-Rule.md) への適合（**色のハードコード禁止**、テーマ変数を使用）
- ライト/ダーク、多言語（簡中/繁中/英/日）の考慮

### 5. 実装ステップ（あなたの計画）
1. …
2. …
3. …

### 6. テストと検証
- 追加する単体/統合テスト
- 手動検証パス

### 7. リスクと代替案
- 既存ユーザーデータ・バックアップ・同期への影響
- この機能をしない場合の軽量な代替

### 8. 承認後に実装する意思は？
（はい / いいえ / アイデアのみ）
```

メンテナは **承認 / 修正要 / お断り / 保留** で返答します。  
**「承認」または「PR 歓迎」と明示された後** に Fork して PR を出してください。

### 3.3 monorepo マップ（「何が変わるか」の参考）

| パス | 役割 |
| ---- | ---- |
| `apps/desktop` | Electron デスクトップ（React + electron-vite） |
| `apps/mobile` | Expo / React Native モバイル |
| `packages/core` | クロスプラットフォームのコアロジック |
| `packages/core-desktop` / `core-mobile` | プラットフォーム固有コア |
| `packages/ai` | AI Provider、Agent、ツール呼び出し |
| `packages/database` / `database-desktop` | libSQL/SQLite + Drizzle |
| `packages/ui` | 共有 UI、テーマ、日記エディター等 |
| `packages/shared` / `store` | ユーティリティ、状態 |

機能変更はしばしば **UI + core + database + 両アプリ** に及びます。Issue では範囲を正直に書いてください。

### 3.4 事前承認が通常必要な変更

（[1-AI-Code-Rule.md §6](../1-AI-Code/1-AI-Code-Rule.md) と同じ）

- 新 npm 依存（特に Native addon、>500KB、新ビルドチェーン）
- DB Schema またはマイグレーション
- 公開 API / IPC の Breaking Change
- 新サードパーティ UI ライブラリ
- CI / リリースパイプラインの変更

## 4. その他の貢献

- **ドキュメント**：誤り修正・開発メモの追加 — PR 歓迎。
- **アイデア・議論**：Issue で歓迎（例：[アイデア](https://github.com/Anson-Trio/BaiShou-Next/issues/new?template=03-interesting.yml) テンプレート）が、ロードマップ採用を意味しません。
- **個人利用の Fork**：AGPLv3 で許可；改変版をネットワークサービスとして提供する場合はライセンスに従いソースを公開してください。

---

## 5. PR がクローズされる主な理由

- 事前承認された Feature Issue なし
- `pnpm ci:check` 未実行または失敗
- テスト不足（trivial でない修正）
- 未承認の依存または UI ライブラリ
- テーマ規約違反（色のハードコード等）
- 範囲が大きすぎる、無関係な変更の混在
